Throughout this document, we use the following conventions.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md531}{}\doxysubsection{\texorpdfstring{Commands}{Commands}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md531}
Any line starting with a dollar sign is a command line. \begin{DoxyVerb}$ command
\end{DoxyVerb}
 The dollar sign indicates the shell prompt and is not to be entered as part of the command.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md532}{}\doxysubsection{\texorpdfstring{Choices}{Choices}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md532}
Several words in curly braces separated by pipe characters indicate a {\bfseries{mandatory choice}}, to be replaced with one of the given words. For example, the line \begin{DoxyVerb}$ echo { WORD1 | WORD2 | WORD3 }
\end{DoxyVerb}
 represents one of the following three commands \begin{DoxyVerb}$ echo WORD1
- or -
$ echo WORD2
- or -
$ echo WORD3
\end{DoxyVerb}
 One or several words in square brackets separated by pipe characters denote an {\bfseries{optional choice}}. It is similar to the mandatory choice, but it can also be omitted entirely.

So the line \begin{DoxyVerb}$ echo [ WORD1 | WORD2 | WORD3 ]
\end{DoxyVerb}
 represents one of the four commands \begin{DoxyVerb}$ echo WORD1
- or -
$ echo WORD2
- or -
$ echo WORD3
- or -
$ echo
\end{DoxyVerb}

{\bfseries{Optional Arguments}} are enclosed in square brackets. \begin{DoxyVerb}[option...]
\end{DoxyVerb}

The following commands will install Open\+SSL to a default system location.

{\bfseries{Danger Zone\+:}} even if you are impatient, please read the following two paragraphs carefully before you install Open\+SSL.

For security reasons the default system location is by default not writable for unprivileged users. So for the final installation step administrative privileges are required. The default system location and the procedure to obtain administrative privileges depends on the operating system. It is recommended to compile and test Open\+SSL with normal user privileges and use administrative privileges only for the final installation step.

On some platforms Open\+SSL is preinstalled as part of the Operating System. In this case it is highly recommended not to overwrite the system versions, because other applications or libraries might depend on it. To avoid breaking other applications, install your copy of Open\+SSL to a different location which is not in the global search path for system libraries.

Finally, if you plan on using the FIPS module, you need to read the Post-\/installation Notes further down.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md540}{}\doxysubsubsection{\texorpdfstring{Unix / Linux / mac\+OS}{Unix / Linux / mac\+OS}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md540}

\end{DoxyVerb}
 depending on the file type, as it is custom on Unix-\/like operating systems.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md541}{}\doxysubsubsection{\texorpdfstring{Open\+VMS}{Open\+VMS}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md541}

To install Open\+SSL to a different location (for example into your home directory for testing purposes) run {\ttfamily Configure} as shown in the following examples.

 Note\+: if you do add options to the configuration command, please make sure you\textquotesingle{}ve read more than just this Quick Start, such as relevant {\ttfamily NOTES-\/\texorpdfstring{$\ast$}{*}} files, the options outline below, as configuration options may change the outcome in otherwise unexpected ways.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md544}{}\doxysection{\texorpdfstring{Configuration Options}{Configuration Options}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md544}
There are several options to {\ttfamily ./\+Configure} to customize the build (note that for Windows, the defaults for {\ttfamily -\/-\/prefix} and {\ttfamily -\/-\/openssldir} depend on what configuration is used and what Windows implementation Open\+SSL is built on. For more information, see the \doxysectlink{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_n_o_t_e_s-_w_i_n_d_o_w_s}{Notes for Windows platforms}{0}.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md545}{}\doxysubsection{\texorpdfstring{API Level}{API Level}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md545}
\begin{DoxyVerb}--api=x.y[.z]
\end{DoxyVerb}
 Build the Open\+SSL libraries to support the API for the specified version. If no-\/deprecated is also given, don\textquotesingle{}t build with support for deprecated APIs in or below the specified version number. For example, adding \begin{DoxyVerb}--api=1.1.0 no-deprecated
\end{DoxyVerb}
 will remove support for all APIs that were deprecated in Open\+SSL version 1.\+1.\+0 or below. This is a rather specialized option for developers. If you just intend to remove all deprecated APIs up to the current version entirely, just specify no-\/deprecated. If {\ttfamily -\/-\/api} isn\textquotesingle{}t given, it defaults to the current (minor) Open\+SSL version.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md546}{}\doxysubsection{\texorpdfstring{Cross Compile Prefix}{Cross Compile Prefix}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md546}
\begin{DoxyVerb}--cross-compile-prefix=<PREFIX>
\end{DoxyVerb}
 The {\ttfamily \texorpdfstring{$<$}{<}PREFIX\texorpdfstring{$>$}{>}} to include in front of commands for your toolchain.

It is likely to have to end with dash, e.\+g. {\ttfamily a-\/b-\/c-\/} would invoke GNU compiler as {\ttfamily a-\/b-\/c-\/gcc}, etc. Unfortunately cross-\/compiling is too case-\/specific to put together one-\/size-\/fits-\/all instructions. You might have to pass more flags or set up environment variables to actually make it work. Android and i\+OS cases are discussed in corresponding {\ttfamily Configurations/15-\/\texorpdfstring{$\ast$}{*}.conf} files. But there are cases when this option alone is sufficient. For example to build the mingw64 target on Linux {\ttfamily -\/-\/cross-\/compile-\/prefix=x86\+\_\+64-\/w64-\/mingw32-\/} works. Naturally provided that mingw packages are installed. Today Debian and Ubuntu users have option to install a number of prepackaged cross-\/compilers along with corresponding run-\/time and development packages for "{}alien"{} hardware. To give another example {\ttfamily -\/-\/cross-\/compile-\/prefix=mipsel-\/linux-\/gnu-\/} suffices in such case.

For cross compilation, you must configure manually. Also, note that {\ttfamily -\/-\/openssldir} refers to target\textquotesingle{}s file system, not one you are building on.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md547}{}\doxysubsection{\texorpdfstring{Build Type}{Build Type}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md547}

This is the default.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md548}{}\doxysubsection{\texorpdfstring{Directories}{Directories}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md548}
\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md549}{}\doxysubsubsection{\texorpdfstring{libdir}{libdir}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md549}
\begin{DoxyVerb}--libdir=DIR
\end{DoxyVerb}
 The name of the directory under the top of the installation directory tree (see the {\ttfamily -\/-\/prefix} option) where libraries will be installed. By default this is {\ttfamily lib}. Note that on Windows only static libraries ({\ttfamily \texorpdfstring{$\ast$}{*}.lib}) will be stored in this location. Shared libraries ({\ttfamily \texorpdfstring{$\ast$}{*}.dll}) will always be installed to the {\ttfamily bin} directory.

Some build targets have a multilib postfix set in the build configuration. For these targets the default libdir is {\ttfamily lib\texorpdfstring{$<$}{<}multilib-\/postfix\texorpdfstring{$>$}{>}}. Please use {\ttfamily -\/-\/libdir=lib} to override the libdir if adding the postfix is undesirable.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md550}{}\doxysubsubsection{\texorpdfstring{openssldir}{openssldir}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md550}
\begin{DoxyVerb}--openssldir=DIR
\end{DoxyVerb}
 Directory for Open\+SSL configuration files, and also the default certificate and key store. Defaults are\+: \begin{DoxyVerb}Unix:           /usr/local/ssl

\end{DoxyVerb}

Don\textquotesingle{}t pin the shared libraries.

By default Open\+SSL will attempt to stay in memory until the process exits. This is so that libcrypto and libssl can be properly cleaned up automatically via an {\ttfamily atexit()} handler. The handler is registered by libcrypto and cleans up both libraries. On some platforms the {\ttfamily atexit()} handler will run on unload of libcrypto (if it has been dynamically loaded) rather than at process exit.

This option can be used to stop Open\+SSL from attempting to stay in memory until the process exits. This could lead to crashes if either libcrypto or libssl have already been unloaded at the point that the atexit handler is invoked, e.\+g. on a platform which calls {\ttfamily atexit()} on unload of the library, and libssl is unloaded before libcrypto then a crash is likely to happen.

Note that shared library pinning is not automatically disabled for static builds, i.\+e., {\ttfamily no-\/shared} does not imply {\ttfamily no-\/pinshared}. This may come as a surprise when linking libcrypto statically into a shared third-\/party library, because in this case the shared library will be pinned. To prevent this behaviour, you need to configure the static build using {\ttfamily no-\/shared} and {\ttfamily no-\/pinshared} together.

Starting with version 1.\+1.\+0, Open\+SSL hides a number of structures that were previously open. This includes all internal libssl structures and a number of EVP types. Accessor functions have been added to allow controlled access to the structures\textquotesingle{} data.

This means that some software needs to be rewritten to adapt to the new ways of doing things. This often amounts to allocating an instance of a structure explicitly where you could previously allocate them on the stack as automatic variables, and using the provided accessor functions where you would previously access a structure\textquotesingle{}s field directly.

Some APIs have changed as well. If the build fails, look carefully at the output. Try to locate and understand the error message. It might be that the compiler is already telling you exactly what you need to do to fix your problem.

There may be reasons for the failure that aren\textquotesingle{}t problems in Open\+SSL itself, for example if the compiler reports missing standard or third party headers.

If the build succeeded previously, but fails after a source or configuration change, it might be helpful to clean the build tree before attempting another build. Use this command\+: \begin{DoxyVerb}$ make clean                                     # Unix
$ mms clean                                      ! (or mmk) OpenVMS
$ nmake clean                                    # Windows
\end{DoxyVerb}
 Assembler error messages can sometimes be sidestepped by using the {\ttfamily no-\/asm} configuration option. See also notes.

Compiling parts of Open\+SSL with gcc and others with the system compiler will result in unresolved symbols on some systems.

For some systems, the Open\+SSL {\ttfamily Configure} script knows what compiler options are needed to generate a library that is suitable for multi-\/threaded applications. On these systems, support for multi-\/threading is enabled by default; use the {\ttfamily no-\/threads} option to disable (this should never be necessary).

On other systems, to enable support for multi-\/threading, you will have to specify at least two options\+: {\ttfamily threads}, and a system-\/dependent option. (The latter is {\ttfamily -\/D\+\_\+\+REENTRANT} on various systems.) The default in this case, obviously, is not to include support for multi-\/threading (but you can still use {\ttfamily no-\/threads} to suppress an annoying warning message from the {\ttfamily Configure} script.)

Open\+SSL provides built-\/in support for two threading models\+: pthreads (found on most UNIX/\+Linux systems), and Windows threads. No other threading models are supported. If your platform does not provide pthreads or Windows threads then you should use {\ttfamily Configure} with the {\ttfamily no-\/threads} option.

For pthreads, all locks are non-\/recursive. In addition, in a debug build, the mutex attribute {\ttfamily PTHREAD\+\_\+\+MUTEX\+\_\+\+ERRORCHECK} is used. If this is not available on your platform, you might have to add {\ttfamily -\/DOPENSSL\+\_\+\+NO\+\_\+\+MUTEX\+\_\+\+ERRORCHECK} to your {\ttfamily Configure} invocation. (On Linux {\ttfamily PTHREAD\+\_\+\+MUTEX\+\_\+\+ERRORCHECK} is an enum value, so a built-\/in ifdef test cannot be used.)\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md699}{}\doxysubsection{\texorpdfstring{Notes on shared libraries}{Notes on shared libraries}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md699}
For most systems the Open\+SSL {\ttfamily Configure} script knows what is needed to build shared libraries for libcrypto and libssl. On these systems the shared libraries will be created by default. This can be suppressed and only static libraries created by using the {\ttfamily no-\/shared} option. On systems where Open\+SSL does not know how to build shared libraries the {\ttfamily no-\/shared} option will be forced and only static libraries will be created.

Shared libraries are named a little differently on different platforms. One way or another, they all have the major Open\+SSL version number as part of the file name, i.\+e. for Open\+SSL 1.\+1.\+x, {\ttfamily 1.\+1} is somehow part of the name.

On most POSIX platforms, shared libraries are named {\ttfamily libcrypto.\+so.\+1.\+1} and {\ttfamily libssl.\+so.\+1.\+1}.

on Cygwin, shared libraries are named {\ttfamily cygcrypto-\/1.\+1.\+dll} and {\ttfamily cygssl-\/1.\+1.\+dll} with import libraries {\ttfamily libcrypto.\+dll.\+a} and {\ttfamily libssl.\+dll.\+a}.

On Windows build with MSVC or using MingW, shared libraries are named {\ttfamily libcrypto-\/1\+\_\+1.\+dll} and {\ttfamily libssl-\/1\+\_\+1.\+dll} for 32-\/bit Windows, {\ttfamily libcrypto-\/1\+\_\+1-\/x64.\+dll} and {\ttfamily libssl-\/1\+\_\+1-\/x64.\+dll} for 64-\/bit x86\+\_\+64 Windows, and {\ttfamily libcrypto-\/1\+\_\+1-\/ia64.\+dll} and {\ttfamily libssl-\/1\+\_\+1-\/ia64.\+dll} for IA64 Windows. With MSVC, the import libraries are named {\ttfamily libcrypto.\+lib} and {\ttfamily libssl.\+lib}, while with MingW, they are named {\ttfamily libcrypto.\+dll.\+a} and {\ttfamily libssl.\+dll.\+a}.

On VMS, shareable images (VMS speak for shared libraries) are named {\ttfamily ossl\$libcrypto0101\+\_\+shr.\+exe} and {\ttfamily ossl\$libssl0101\+\_\+shr.\+exe}. However, when Open\+SSL is specifically built for 32-\/bit pointers, the shareable images are named {\ttfamily ossl\$libcrypto0101\+\_\+shr32.\+exe} and {\ttfamily ossl\$libssl0101\+\_\+shr32.\+exe} instead, and when built for 64-\/bit pointers, they are named {\ttfamily ossl\$libcrypto0101\+\_\+shr64.\+exe} and {\ttfamily ossl\$libssl0101\+\_\+shr64.\+exe}.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md700}{}\doxysubsection{\texorpdfstring{Notes on random number generation}{Notes on random number generation}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md700}
Availability of cryptographically secure random numbers is required for secret key generation. Open\+SSL provides several options to seed the internal CSPRNG. If not properly seeded, the internal CSPRNG will refuse to deliver random bytes and a "{}\+PRNG not seeded error"{} will occur.

The seeding method can be configured using the {\ttfamily -\/-\/with-\/rand-\/seed} option, which can be used to specify a comma separated list of seed methods. However, in most cases Open\+SSL will choose a suitable default method, so it is not necessary to explicitly provide this option. Note also that not all methods are available on all platforms. The FIPS provider will silently ignore seed sources that were not validated.

I) On operating systems which provide a suitable randomness source (in form of a system call or system device), Open\+SSL will use the optimal available method to seed the CSPRNG from the operating system\textquotesingle{}s randomness sources. This corresponds to the option {\ttfamily -\/-\/with-\/rand-\/seed=os}.

II) On systems without such a suitable randomness source, automatic seeding and reseeding is disabled ({\ttfamily -\/-\/with-\/rand-\/seed=none}) and it may be necessary to install additional support software to obtain a random seed and reseed the CSPRNG manually. Please check out the manual pages for {\ttfamily RAND\+\_\+add()}, {\ttfamily RAND\+\_\+bytes()}, {\ttfamily RAND\+\_\+egd()}, and the FAQ for more information.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md701}{}\doxysubsection{\texorpdfstring{Notes on assembler modules compilation}{Notes on assembler modules compilation}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2_i_n_s_t_a_l_l_autotoc_md701}

(\texorpdfstring{$\ast$}{*}) Even though AVX512 support was implemented in llvm 3.\+6, prior to version 7.\+0 an explicit -\/march flag was apparently required to compile assembly modules. But then the compiler generates processor-\/specific code, which in turn contradicts the idea of performing dispatch at run-\/time, which is facilitated by the special variable {\ttfamily OPENSSL\+\_\+ia32cap}. For versions older than 7.\+0, it is possible to work around the problem by forcing the build procedure to use the following script\+: \begin{DoxyVerb}#!/bin/sh
exec clang -no-integrated-as "$@"
\end{DoxyVerb}
 instead of the real clang. In which case it doesn\textquotesingle{}t matter what clang version is used, as it is the version of the GNU assembler that will be checked.
