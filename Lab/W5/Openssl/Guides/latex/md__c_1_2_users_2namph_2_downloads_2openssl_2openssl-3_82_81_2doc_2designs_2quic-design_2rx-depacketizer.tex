\chapter{RX depacketizer}
\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer}{}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer}\index{RX depacketizer@{RX depacketizer}}
This component takes a QUIC packet and parses the frames contained therein, to be forwarded to appropriate other components for further processing.

In the \href{https://github.com/openssl/openssl/blob/master/doc/designs/quic-design/quic-overview.md}{\texttt{ overview}}, this is called the "{}\+RX Frame Handler"{}. The name "{}\+RX depacketizer"{} was chosen to reflect the kinship with the \href{https://github.com/openssl/openssl/pull/18570}{\texttt{ TX packetizer}}.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md461}{}\doxysection{\texorpdfstring{Main structures}{Main structures}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md461}
\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md462}{}\doxysubsection{\texorpdfstring{Connection}{Connection}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md462}
Represented by an {\ttfamily QUIC\+\_\+\+CONNECTION} object, defined in \href{../../../include/internal/quic_ssl.h}{\texttt{ {\ttfamily include/internal/quic\+\_\+ssl.\+h}}}.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md463}{}\doxysubsection{\texorpdfstring{Stream}{Stream}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md463}
Represented by an {\ttfamily QUIC\+\_\+\+STREAM} object (yet to be defined).\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md464}{}\doxysubsection{\texorpdfstring{Packets}{Packets}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md464}
Represented by the {\ttfamily OSSL\+\_\+\+QRX\+\_\+\+PKT} structure, defined in {\ttfamily \doxylink{quic__record__rx_8h_source}{include/internal/quic\+\_\+record\+\_\+rx.\+h}} in \href{https://github.com/openssl/openssl/pull/18949}{\texttt{ QUIC Demuxer and Record Layer (RX+\+TX)}}.\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md465}{}\doxysection{\texorpdfstring{Interactions}{Interactions}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md465}
The RX depacketizer receives a packet from the QUIC Read Record Layer, and then processes frames in two phases\+:


\begin{DoxyEnumerate}
\item Collect information for the ACK Manager
\item Pass frame data
\end{DoxyEnumerate}\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md466}{}\doxysubsection{\texorpdfstring{Other components}{Other components}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md466}
There are a number of other components that the RX depacketizer wants to interact with\+:


\begin{DoxyItemize}
\item \href{https://github.com/openssl/openssl/pull/18564}{\texttt{ ACK manager}}
\item Handshake manager, which is currently unspecified. It\textquotesingle{}s assumed that this will wrap around what is called the "{}\+TLS Handshake Record Layer"{}, in the \href{https://github.com/openssl/openssl/blob/master/doc/designs/quic-design/quic-overview.md}{\texttt{ overview}}
\item Session manager, which is currently unspecified for QUIC, but may very well be the existing {\ttfamily SSL\+\_\+\+SESSION} functionality, extended to fit QUIC purposes.
\item Flow control, which is currently unspecified. In the \href{https://github.com/openssl/openssl/blob/master/doc/designs/quic-design/quic-overview.md}{\texttt{ overview}}, it\textquotesingle{}s called the "{}\+Flow Controller And Statistics Collector"{}
\item Connection manager, which is currently unspecified. In the \href{https://github.com/openssl/openssl/blob/master/doc/designs/quic-design/quic-overview.md}{\texttt{ overview}}, there\textquotesingle{}s a "{}\+Connection State Machine"{} that the "{}\+RX Frame Handler"{} isn\textquotesingle{}t talking directly with, so it\textquotesingle{}s possible that the Connection manager will turn out to be the Handshake manager.
\item Stream SSL objects, to pass the stream data to.
\end{DoxyItemize}\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md467}{}\doxysubsection{\texorpdfstring{Read and process a packet}{Read and process a packet}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md467}
Following how things are designed elsewhere, the depacketizer is assumed to be called "{}from above"{} using the following function\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\ C}
\DoxyCodeLine{\_\_owur\ int\ ossl\_quic\_depacketize(QUIC\_CONNECTION\ *connection);}

\end{DoxyCode}


This function would create an {\ttfamily OSSL\+\_\+\+QRX\+\_\+\+PKT} and call the QUIC Read Record Layer with a pointer to it, leaving it to the QUIC Read Record Layer to fill in the data.

This uses the {\ttfamily ossl\+\_\+qrx\+\_\+read\+\_\+pkt()} packet reading function from \href{https://github.com/openssl/openssl/pull/18949}{\texttt{ QUIC Demuxer and Record Layer (RX+\+TX)}}. (the {\ttfamily OSSL\+\_\+\+QRX\+\_\+\+PKT} structure / sub-\/structure needs to be extended to take an {\ttfamily \doxylink{struct_o_s_s_l___t_i_m_e}{OSSL\+\_\+\+TIME}}, possibly by reference, which should be filled in with the packet reception time)\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md468}{}\doxysubsection{\texorpdfstring{Collect information for the \href{https://github.com/openssl/openssl/pull/18564}{\texttt{ ACK manager}}}{Collect information for the \href{https://github.com/openssl/openssl/pull/18564}{\texttt{ ACK manager}}}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md468}
This collects appropriate data into a {\ttfamily QUIC\+\_\+\+ACKM\+\_\+\+RX\+\_\+\+PKT} structure\+:


\begin{DoxyItemize}
\item The packet number ({\ttfamily packet-\/\texorpdfstring{$>$}{>}packet\+\_\+number})
\item The packet receive time ({\ttfamily received})
\item The packet space, which is always\+:
\begin{DoxyItemize}
\item {\ttfamily QUIC\+\_\+\+PN\+\_\+\+SPACE\+\_\+\+INITIAL} when {\ttfamily packet-\/\texorpdfstring{$>$}{>}packet\+\_\+type == pkt\+\_\+initial}
\item {\ttfamily QUIC\+\_\+\+PN\+\_\+\+SPACE\+\_\+\+HANDSHAKE} when {\ttfamily packet-\/\texorpdfstring{$>$}{>}packet\+\_\+type == pkt\+\_\+handshake}
\item {\ttfamily QUIC\+\_\+\+PN\+\_\+\+SPACE\+\_\+\+APP} for all other packet types
\end{DoxyItemize}
\item The ACK eliciting flag. This is calculated by looping through all frames and noting those that are ACK eliciting, as determined from Table 1 below)
\end{DoxyItemize}\hypertarget{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md469}{}\doxysubsection{\texorpdfstring{Passing frame data}{Passing frame data}}\label{md__c_1_2_users_2namph_2_downloads_2openssl_2openssl-3_82_81_2doc_2designs_2quic-design_2rx-depacketizer_autotoc_md469}

\mbox{[}\texorpdfstring{$^\wedge$}{\string^}1\mbox{]}\+: This creates and populates an {\ttfamily QUIC\+\_\+\+ACKM\+\_\+\+ACK} structure, then calls {\ttfamily QUIC\+\_\+\+ACKM\+\_\+on\+\_\+rx\+\_\+ack\+\_\+frame()}, with the appropriate context ({\ttfamily QUIC\+\_\+\+ACKM}, the created {\ttfamily QUIC\+\_\+\+ACKM\+\_\+\+ACK}, {\ttfamily pkt\+\_\+space} and {\ttfamily rx\+\_\+time}) \mbox{[}\texorpdfstring{$^\wedge$}{\string^}2\mbox{]}\+: Immediately terminates the appropriate receiving stream {\ttfamily QUIC\+\_\+\+STREAM} object. This includes discarding any buffered application data. For a stream that\textquotesingle{}s send-\/only, the error {\ttfamily STREAM\+\_\+\+STATE\+\_\+\+ERROR} is raised, and the {\ttfamily QUIC\+\_\+\+CONNECTION} object is terminated. \mbox{[}\texorpdfstring{$^\wedge$}{\string^}3\mbox{]}\+: Immediately terminates the appropriate sending stream {\ttfamily QUIC\+\_\+\+STREAM} object. For a stream that\textquotesingle{}s receive-\/only, the error {\ttfamily STREAM\+\_\+\+STATE\+\_\+\+ERROR} is raised, and the {\ttfamily QUIC\+\_\+\+CONNECTION} object is terminated. \mbox{[}\texorpdfstring{$^\wedge$}{\string^}4\mbox{]}\+: The frame payload (Stream Data) is passed as is to the {\ttfamily QUIC\+\_\+\+STREAM} object, along with available metadata (offset and length, as determined to be available from the lower 3 bits of the frame type). \mbox{[}\texorpdfstring{$^\wedge$}{\string^}5\mbox{]}\+: The details of what flow control will need are yet to be determined \mbox{[}\texorpdfstring{$^\wedge$}{\string^}6\mbox{]}\+: I imagine that {\ttfamily max\+\_\+streams} and {\ttfamily streams\+\_\+blocked} concern a Connection manager before anything else. \mbox{[}\texorpdfstring{$^\wedge$}{\string^}7\mbox{]}\+: I imagine that path challenge/response concerns a Connection manager before anything else. \mbox{[}\texorpdfstring{$^\wedge$}{\string^}8\mbox{]}\+: We have no idea what extension frames there will be.
